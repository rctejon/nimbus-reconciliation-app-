# Design Writeup

The reconciliation engine uses a two-phase matching strategy: it first attempts an exact ID match by comparing each transaction's `transaction_id` against settlement `transaction_reference` fields, then falls back to fuzzy matching that scores candidates by amount proximity (60% weight) and date proximity (40% weight) within a 2% tolerance window and 7-day date range. All monetary amounts are converted to USD using a static exchange rate table (USD=1.0, MXN=19.5, BRL=4.9, COP=4200.0) before comparison, and cross-currency pairs receive an additional 1% tolerance buffer to account for conversion imprecision. The engine detects duplicate settlements by grouping on `transaction_reference` and flagging extras, while batch reconciliation aggregates settlement totals per `batch_id` and compares them against matched expected amounts to surface batch-level discrepancies. An alert system classifies issues by priority: HIGH for single discrepancies exceeding $500 USD or currency slippage above 5%, MEDIUM for unmatched records older than 14 days, and LOW for fee-related differences.

**Tradeoffs:** SQLite was chosen for zero-configuration local development, though a production deployment would use PostgreSQL for concurrency and durability. Static exchange rates simplify the implementation but cannot reflect real-time FX fluctuations; a production system would integrate a live rate API with historical snapshots. The 2% amount tolerance balances catching legitimate processor fee variations against avoiding false-positive matches, and the in-memory matching approach loads all records before reconciling, which is practical for the current dataset size but would require streaming or partitioned processing at scale.

**Future improvements** would include real-time settlement webhooks for continuous reconciliation, historical exchange rate snapshots for accurate retroactive matching, configurable tolerance thresholds per processor or currency pair, role-based access control, a comprehensive audit log, and an async job queue (e.g. Celery or arq) to handle large reconciliation runs without blocking the API.
